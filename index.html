Birthday Paradox Simulation

What is the Birthday Paradox? The Birthday Paradox is a well know probability problem that states if you have a group of 23 people, there is a 50% chance that two of them will share the same birthday.

What does this project do? This project simulates the Birthday paradox by randomly generating birthdays for different group sizes, ranging from 5 to 100 people in increments of 5. Each group undergoes 20 trials, and the program calculates the percentage of trial where at least two people share the same birthday. The results are displayed showing how the probability increases as the group size grows.

Functional Requirements

  • Generates random birthdays while ensuring validity (Example: no February 30th).
  • Simulates groups of 5 to 100 people in increments of 5.
  • Runs 20 trials per group size to compute the probability of shared birthdays.
  • Uses a single Random object to ensure consistent seeding.
  • Outputs results in a structured table format.

Implementation Details (My Contribution)

In this project, I was responsible for:

  • Implementing the Experiment.java class, which runs multiple trials to determine the probability of duplicate birthdays within a group.
  • Developing the Trial.java class to simulate individual birthday matching scenarios. This class generates random birthdays and checks for duplicates.
  • Writing unit tests in TrialTest.java using JUnit and Mockito to verify that trials correctly detect duplicate birthdays and generate the expected number of birthdays.
  • Applying test-driven development to verify that the simulation logic worked as expected before implementation.

This project helped me improve my Java coding in many ways. In particular I was able to practice and get more comfortable with Test Driven development. I was able to get tons of practice with testing, such as writing Unit tests and Mocking them. This process also improved my debugging skills along with getting comfortable using Github with partners to improve code.

Elevator Simulation Project

Project Title: Elevator Simulator

This project involves building an elevator simulation system that models multiple types of elevators servicing a building with varying rules. The different types of elevators that were implemented were Standard, Freight, Penthouse, and Express, each has its own movement and call behavior. The simulation operates through rounds, where a scheduler assigns elevator calls, and elevators move one floor per round based on their logic.

Functional Requirements:

  • Basic Simulator Mechanics:
    • Loads calls from a file.
    • Creates elevators with the appropriate types and quantities.
    • Assigns calls to elevators using the Scheduler.
    • Simulates elevator movement floor by floor.
  • Correct Call Assignment:
    • Calls are assigned based on the behavior of the selected algorithm (Simple or Complex).
    • Elevators only service calls they are allowed to, based on elevator type.
  • Elevator Call Processing:
    • Elevators pick up passengers at the call’s start floor.
    • Drop them off at their destination floor.
    • Track the completion status of each call.
  • Correct Termination Condition:
    • Simulation ends only when all calls are completed and elevators are back on the ground floor.
  • Output Accuracy:
    • Accurately displays each elevator’s status (floor, direction, passengers).
    • Shows unassigned calls and clearly reflects completion of all simulation rounds.

Design Decisions:

We designed the project using object-oriented principles like inheritance and abstraction to represent different elevator types. Test-driven development was used throughout the process to ensure functionality. Each elevator type was given a specific behavior model, making the simulation flexible and scalable.

My Contributions:

  • Implemented the core Simulator class that manages the entire simulation loop and termination logic.
  • Wrote unit tests using JUnit and Mockito to verify that elevators moved correctly and handled calls properly, along with tests for all classes that I worked on.
  • Worked on the Scheduler classes, more specifically ComplexScheduler and SimpleScheduler, to dispatch calls to elevators based on their type and current state.
  • Wrote the Building class, which manages configurations of elevators in the building.
  • Engaged in paired programming with a partner using GitHub for version control and branching strategies.
  • Wrote the Call class that stores information about elevator calls.

What I Learned:

This project taught me how to apply test-driven development in a large codebase and improved my ability to write testable Java code. It also strengthened my collaboration and debugging skills, especially when integrating with my partner's code in real-time using Git.
